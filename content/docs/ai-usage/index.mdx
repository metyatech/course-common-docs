# 生成AIの利用

## 生成AIとは

**生成AI（Generative AI）** とは、質問や指示を普通の言葉で入力すると、文章・コード・画像などを自動で作ってくれるAIのことです。

代表的なサービスには次のようなものがあります。

- **ChatGPT**（OpenAI） — 文章やコードの生成・質問への回答など、幅広い用途に使える対話型AI
- **Gemini**（Google） — Google が提供する対話型AI。Google 検索との連携が特徴
- **Claude**（Anthropic） — 長い文章の読解や丁寧な説明が得意な対話型AI
- **GitHub Copilot**（GitHub） — プログラミング向けのAI。エディタ上でコードを提案してくれるほか、チャットで質問もできる

いずれも、**質問や指示を出すと、AIが回答やコードを生成してくれる** という点は共通です。

さらに最近では、**AIエージェント**（Claude Code、Codex、GitHub Copilot の Agent モードなど）と呼ばれるツールも登場しています。こちらはチャットでの会話だけでなく、**ファイルの編集やコマンドの実行までAIが自動で行う** のが特徴です。便利な反面、一度に大量の変更が行われるため、**自分で内容を把握する意識がより大切** になります。

どちらを使う場合でも、このページのルールは同じように当てはまります。この授業では、こうしたサービスを実際に使いながら学習を進めていきます。

---

「生成AIは使っていいの？」と迷ったら、まずは次の3つだけ覚えてください。

:::important[先に結論]
- **生成AI** は原則使ってOKです。
- **手書きできること** より、*コードが何をしていて、どこを直せばよいか* を自分で説明できることが大切です。
- <u>自分で扱えないものを、そのまま提出する</u> のはNGです。
:::

この授業では、**授業中も試験中も**、生成AIを学習や作業の **道具** として使ってかまいません。

## 使ってよい使い方

**OKな使い方** は、*理解する*・*下書きを作る*・*比べて学ぶ* の3つです。

### 分からないことを理解するために使う

:::tip[OK例]
- **エラーを理解したい**: 「このエラー文の意味を、やさしい言葉で説明して」
- **コードの目的を知りたい**: 「このコードは何を実現していて、どこが重要か説明して」
- **説明を分かりやすくしたい**: 「この文章を、高校生にも分かる言い方に直して」
:::

:::note[なぜ？]
- **理由**: 意味をつかむために使うなら、AIは *答えを丸ごと出す機械* ではなく、**理解を助ける道具** になります。
- **たとえば**: エラーの意味をAIに聞いて理解したうえで、自分でコードを直す。これなら **何をするコードか** と **どこが重要か** を最終的に判断しているのは自分です。
:::

### 下書きを作ってもらい、自分で直す

:::tip[OK例]
- **見出しの案がほしい**: 「レポートの見出し案を3つ出して」
- **たたき台がほしい**: 「このコードのたたき台を作って」
- **別案を見たい**: 「別の書き方の例を2つ見せて」
:::

:::note[なぜ？]
- **理由**: 下書きは *完成品* ではなく、**出発点** だからです。
- **たとえば**: AIが作ったたたき台には、<u>想定外の動き</u> や不要な処理が入ることがあります。自分で読み直して、**何をするコードか** と **どこを変えたか** を説明できるところまで仕上げて、はじめて自分の学びになります。
:::

### 比べて学ぶ

:::tip[OK例]
- **別解と比較したい**: 「この書き方と別解の違いを教えて」
- **説明の弱いところを知りたい**: 「この説明のどこが分かりにくいか教えて」
- **改善点を知りたい**: 「このコードの改善点を3つ挙げて」
:::

:::note[なぜ？]
- **理由**: 比較することで、**1つの答えをうのみにせず** 自分で考えられるようになります。
- **たとえば**: 自分のコードと別解を見比べると、*なぜこの書き方の方が読みやすいのか* が分かります。こうした気づきは次の課題にも活きます。
:::

## 使ってはいけない使い方

**NGな使い方** は、*説明なし*・*確認なし*・*情報漏えい* です。

### 動作や直し方を説明できないまま提出する

:::caution[NG例]
- **そのまま提出**: AIが作ったコードを、<u>何をするものか確認せずに</u> そのまま提出する。
- **直し方が言えない**: 「どこを直せばよいか」を自分で言えないまま提出する。
- **説明できない文章**: AIが作った文章を、<u>自分の言葉で説明できないまま</u> 提出する。
:::

:::note[なぜ？]
- **理由**: 提出物は、*自分が責任をもつ成果物* だからです。
- **最低限必要なこと**: ここでいう理解は、全行を手書きできることではありません。**何をするものか**、**どこが重要か**、**壊れたらどこを見て直すか** が言えないと、自分で扱える状態とは言えません。
:::

### AIの答えを確認せずに信じる

:::caution[NG例]
- **未実行で使う**: <u>実行していないコード</u> を、そのまま正しいと思って使う。
- **無検証で信じる**: AIが書いた説明を、教科書や授業内容と照らし合わせずにうのみにする。
:::

:::note[なぜ？]
- **理由**: 生成AIは便利ですが、*もっともらしい間違い* も普通に出します。
- **たとえば**: 存在しない関数名を書いたり、授業で扱っていない仕様を混ぜたりすることがあります。見た目が自然でも、中身が間違っていることがあります。だから **どこを確認すればよいか** を自分で押さえておく必要があります。
:::

### 入力してはいけない情報を入れる

:::caution[NG例]
- **ログイン情報**: 学校アカウントの情報やパスワードを入力する。
- **未公開情報**: 未公開の成績情報や個人情報を入力する。
:::

:::note[なぜ？]
- **理由**: 一度入力した情報は、*自分の手元だけで完結しない* 可能性があるからです。
- **たとえば**: 「ログインできないから見て」と言って、<u>IDやパスワードをそのまま入力する</u> のは危険です。困ったときは、**情報を伏せた状態** で質問してください。
:::

## 試験中の使い方

**試験中** も考え方は同じです。<u>AIの答えをそのまま写すだけ</u> は認めません。

:::note[基本ルール]
- **原則**: 中間試験・定期試験でも、担当教員から別の指示がない限り、検索と生成AIの利用を認めます。
- **優先**: 試験ごとに個別ルールがある場合は、*その指示が最優先* です。
:::

:::tip[OK例]
- **候補を出してもらう**: 「このコードのエラー候補を3つ挙げて」
- **見落としを確認する**: 「この説明で見落としている点がないか確認して」
- **修正方針を整理する**: 「このコードが何をしていて、どこを直せばよいか整理して」
:::

:::caution[NG例]
- **丸ごと作らせる**: 「この問題の答えをそのまま全部作って」
- **そのまま写す**: AIが出した答えを、<u>自分で確認せずに</u> そのまま答案に書く。
:::

:::note[なぜ？]
- **理由**: この授業で見たいのは、AIを使ったかどうかよりも、**AIを使っても自分で判断できるか** です。
- **たとえば**: AIにエラー候補を出してもらい、自分で原因を絞って直すなら、それは *丸投げ* ではなく判断です。反対に、AIの答えをそのまま写すだけでは、**動作を理解し、修正方針を持てているか** が確認できません。
:::

## 迷ったときは、この3つで判断してください

:::important[チェックポイント]
1. **何をするものか** を、自分の言葉で説明できますか。
2. **どこを見てどう直すか** を、説明できますか。
3. **入れてはいけない情報** を含んでいませんか。
:::

この3つにすべて「はい」と言えるなら、**かなり安全な使い方** です。*迷ったときは、ここに戻って確認してください。*
